# Know-How智能体自进化方案
**基于记忆与自我反思的输入-输出映射迭代架构**

## 一、 项目背景与目标
在复杂的业务实务领域，用户输入（提问或任务指令）往往具有较强的专业性、边界模糊性及上下文依赖性。本项目旨在构建一个**自进化智能体（Agent）系统**，将泛化的用户输入精准映射至标准化的业务输出或执行动作。

针对**“无海量结构化训练数据”的现实痛点，系统摒弃了传统的全量模型微调（Fine-Tuning）方案，采用“基于记忆的自我反思与技能发现（Memory-based Self-Reflection and Skill Discovery）”**范式。通过输入少量高质量正样本，让智能体在“推理-试错-反思-修正”的闭环中，自发学习并持续迭代业务处理的 Know-How（Skill规则），最终实现高精度、可解释的小样本自动化任务决策。

## 二、 核心痛点与解决思路架构
在复杂任务规则的自动迭代中，系统极易陷入“灾难性遗忘”（修正新错时破坏了旧规则）、“规则文本膨胀”（提示词过长导致注意力丢失）以及“推理路径幻觉”（复杂推理时中途跑偏）三大陷阱。

为此，本系统设计了四大核心技术机制进行防御与增强：
1. **防范灾难性遗忘**：引入“自动化回归验证（Regression Testing）”机制，确保新提取的Know-How不破坏既有能力。
2. **防范规则膨胀**：采用“toml渐进式加载（基础规则+边缘条件动态加载）”，保持Prompt精简。
3. **防范推理跑偏**：引入“带有回溯剪枝的多路径探索（Multi-path Exploration & Backtracking）”，提升复杂决策的容错率。
4. **保障逻辑严密性**：设计结构化的“智能体反思模板（Reflection Prompting）”，规范化经验总结。

## 三、 系统核心模块设计

### 3.1 双轨制记忆管理模块 (Dual-Track Memory System)
每个核心业务场景/技能（Skill）均配备独立的记忆库，分为语义记忆与情景记忆，避免单一Skill文本过度膨胀。
* **触发机制：**
  * **用户反馈**：主要从微观视角，对每一次用户的session做正反向的【知识】提炼总结经验。（更通识的知识）
  * **主动总结**：主要从全局视角，总结某类工具或策略调用的【经验】。
* **知识记忆（Knowledge Memory / Skill Rules）：**
  * **形态**：以业务场景为对象的向量库，每次推理前召回 top-n 作为外部知识补充、top-n 相关的成功/失败案例。
  * **内容**：大模型本身存在的知识盲区，对智能体基于正反案例的边缘知识总结提炼。
* **任务记忆（Episodic Memory / Dynamic RAG）：**
  * **形态**：以session为分割的长程任务草稿本文件夹。
  * **内容**：运行时的执行日志、中间推理状态缓存（memory.md）。
  * **更新频率**：高。每次智能体推理需要的日志草稿工作区域。

### 3.2 多路径探索与回溯机制 (Multi-path Exploration & Backtracking)
放弃单一的贪婪式线性推理，在面对复杂或低置信度问题时采用概率图探索机制。
* **分叉探索（Branching）**：在推理关键节点，若智能体对当前推导方向的置信度低于阈值，则同时保留 Top-2 候选解决思路并行探索。
* **反思回溯（Backtracking）**：当某一条推理分支深入后发现“条件缺失”或“语义冲突”时，触发回溯机制，剪枝当前错误路径，重新选择其他分支，提高整体推理容错率。
* **样本回流**：每条探索路径的成功与失败结论均作为正/负样本，用于后续的记忆沉淀与技能更新。

### 3.3 自动化回归测试闭环 (Regression CI/CD for Agent Skills)
这是保障智能体能够“稳步向前进化”而非“左右横跳”的基石模块。
* **触发条件**：智能体在训练/运行中遇到处理失败的样本（输出不符合预期），通过反思生成了新版业务规则 `Skill_V_Next`。
* **误差回传**：在推理纠错阶段，针对生成了错误输出的样本，修正对应 Know-How 的误差；若发现正样本的判断依据存在偶然性，也同步提炼修正。
* **回归沙盒（Sandbox Testing）**：
  * 系统自动提取该场景下历史已处理正确的**基准测试集（Baseline Subset）**。
  * 使用 `Skill_V_Next` 重新对基准测试集进行推理。
* **准入控制（Gating logic）**：
  * 若准确率未下降，且成功解决了新错误样本：`Skill_V_Next` 正式替换旧版本，完成迭代。
  * 若导致历史正样本报错（打地鼠效应）：拒绝更新，将“报错的旧样本”和“报错的新样本”共同发给反思模块，要求其生成兼容两者的 `Skill_V_Next_2`。如此往复。
* **训推异步**：对一批样本先做完所有推理任务，生成该版本完整误差回传样本，再做Know-How修正，提供版本控制，防止迭代不收敛。

### 3.4 结构化自我反思框架 (Structured Reflection Prompting)
在迭代Skill时，严禁模型自由发挥，必须按照严格的思维链（CoT）输出标准化JSON，确保经验总结的高质量。
**结构化输出约束包含：**
1. **"Error_Attribution" (错误归因)**：分析旧Skill为何导致误判/输出错误（如：概念重合、特征未覆盖、边界条件缺失）。
2. **"Feature_Extraction" (特征提取)**：提炼当前错误样本中具有区分度的“核心业务本质/关键要素”。
3. **"Conflict_Check" (冲突检查)**：新提取的特征是否与当前规则库中的其他逻辑发生矛盾？
4. **"Skill_Patch" (补丁生成)**：以“若满足[条件A]，则执行[输出B]”的最简逻辑，生成对旧Skill的增量补充或修订。

## 四、 智能体运行工作流 (Workflow)

整个系统分为**“技能学习迭代态（Training/Iteration Phase）”**与**“线上推理服务态（Inference Phase）”**。

### 4.1 技能学习迭代流 (基于少量正样本)
1. **样本输入**：系统逐条读取带有正确【输入-期望输出】标注的正样本。
2. **业务推理**：智能体加载当前规则（`Skill_V_Current`），对样本进行前向推导。
3. **结果校验决策**：
   * **推导一致**：说明当前规则已掌握该样本。将样本存入向量库（情景记忆区），结束该样本处理。
   * **推导不一致**：记录出错的环节和对应的错误偏差。
4. **反思与回归闭环**：
   * 触发 **3.4结构化反思**，提取Know-how，生成 `Skill_V_Next`。
   * 触发 **3.3回归测试**，验证 `Skill_V_Next` 对历史基准集的影响。
   * 通过验证后，固化更新业务Skill，进入下一个样本的学习。

### 4.2 线上推理服务流
1. 用户输入业务问题或任务指令。
2. 智能体对输入进行向量化解析，召回关联的**“知识记忆”**（外部事实+成功/失败案例）。
3. 动态加载当前最优版本的 `Skill_V_Current`。
4. 结合多路径探索机制（必要时回溯），执行推理，最终输出高可靠的业务结论或执行动作，并附带判别解释。

## 五、 项目实施路径图 (Roadmap)
考虑到系统的复杂性，建议分四个阶段（Phases）逐步落地：
* **Phase 1: 冷启动与脚手架搭建**
  * 由业务专家人工撰写核心场景的基线规则（Baseline Skills），避免模型初期发散。
  * 搭建基础的LLM调用链路与向量数据库（记忆库）框架。
* **Phase 2: MVP原型闭环验证**
  * 构建一个小规模高质量测试集（例如200条典型样本）。
  * 选取核心业务场景，重点跑通“推理 -> 错判 -> 反思 -> 回归测试 -> 更新Skill”的全自动闭环。
  * 建立准确率（Accuracy）监控看板，观察迭代曲线是否收敛。
* **Phase 3: 架构增强与全量推演**
  * 引入多路径探索与回溯机制。
  * 接入情景记忆（RAG案例库）。
  * 导入真实业务线样本，进行智能体自动化演进，扩展业务覆盖面。
* **Phase 4: 专家干预与上线评估**
  * 开发“后台可视化管控面板”，允许专家查看当前沉淀的Skill规则文本，并支持人工一键微调与干预。
  * 开展黑盒盲测，评估输出准确率及推理可解释性，完成上线。

## 六、 预期业务价值 (Business Value)
1. **降本增效**：彻底打破传统NLP模型需要数万级标注数据的限制，仅需数百个高质量正样本即可完成冷启动，**节约90%以上的数据标注成本**。
2. **业务资产沉淀**：大模型在迭代中写出的规则（Skill）是自然语言，这意味着系统运行时**自动为您沉淀了一套“人类可读、极其详尽的业务操作指南/SOP”**，这是极为珍贵的数据资产。
3. **极高鲁棒性**：面对频繁变更的业务规则或新政策，系统不需要重训模型。只需喂入几个包含新规则的测试样本，智能体即可自动完成内部规则库的热更新与进化。
